Story time

That time I was blocked on a simple problem
My function was incorrectly returning an array (a collection of data, each item separated by commas) where each piece of data was unknown/undefined.
I trialed my code by outputting the problem array at every step throughout my code logic so I could track where it was going wrong using console.log. I didn't spend time in understanding the error message of 'undefined', and I felt like the way I'd written my code was correct to the point I would run my code multiple times in disbelief that I was still getting it wrong!
After tracking my code throughout each step of my logic, it became clear that the code which outputs my solution needed to be assigned to a variable name and then called upon.
I've learnt for next time to read an understand the error message your computer is providing, because it's literally telling you what is wrong and where it's going wrong!

That time a elegantly solved a problem
There was a challenge which required me remove all odd numbers from a collection of data. This problem was elegantly solved because:
1. I used a method called .filter which achieves the same output as using another method but using less code
2. Short-hand function terminology was used which cuts out extra lines of code
3. The smarts of my code within the function was written out using (I believe) as little code as possible

This made me feel awesome because not only was I grasping the challenge and understanding the process in achieving the desired output, but writing precise code without any fluff.

Problem solving techniques

These are the steps I take in order to solve a problem, using recommended techniques:
  1. Read the question and understand it
  2. Break down the question in laymans terms using Pseudocode, with each comment line representing a chunk of code to write. This allows you to break down a big scary question into manageable bite size peices
  3. Under each comment, write your code and try something so it achieves what you've spelled out in your pseudocode. Remember you're only trying to achieve the desired output, the code doesn't need to be elegant, it just needs to work.
  4. If you encounter a situation where you understand the output but don't have the method required memorised in your toolkit, Google what you're trying to achieve and see whether there is an inbuilt method you can utilise.
  5. Generally at this stage you're 90% there and you can't figure out the last steps to tie out your code to complete the question. In this instance use console logging. Console log your inputs throughout the start, middle and end of your function so you can understand what the output of your function is achieiving throughout your code. You will often find minor mistakes to the syntax of your code which will get you across the line!
  6. If console logging doesn't work and Google doesn't help, time to ask your peers or coaches for help so they can guide you through the problem and act as a sounding board as you talk them through your logic.
  7. Once you've successfully answered the question, improve your process with reflection, go through your completed code and see whether there is a way to enhance, simplify or improve the readability.
